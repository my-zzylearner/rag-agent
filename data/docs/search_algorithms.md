# 搜索排序算法详解

## 信息检索基础

信息检索（Information Retrieval）的核心任务是：给定一个查询（Query），从文档集合中找出最相关的文档，并按相关性排序。

现代搜索系统通常分为两阶段：
1. **召回（Retrieval）**：从海量文档中快速筛选出候选集（几百~几千条）
2. **排序（Ranking）**：对候选集精细排序，选出最终结果

## 经典算法：BM25

BM25（Best Match 25）是最经典的关键词检索算法，是 TF-IDF 的改进版，至今仍是全文搜索的基准算法。

### 公式
```
Score(D, Q) = Σ IDF(qᵢ) × [TF(qᵢ,D) × (k1+1)] / [TF(qᵢ,D) + k1 × (1 - b + b × |D|/avgdl)]
```

其中：
- TF(qᵢ, D)：词 qᵢ 在文档 D 中的词频
- IDF(qᵢ)：逆文档频率，衡量词的区分度
- |D|：文档长度，avgdl：平均文档长度
- k1（通常 1.2~2.0）：控制词频饱和度
- b（通常 0.75）：控制文档长度归一化程度

### 特点
- 优点：计算快，无需训练，可解释性强，对精确匹配效果好
- 缺点：无法理解语义，"汽车"和"轿车"被视为完全不同的词

## 向量检索（语义检索）

将查询和文档都转换为向量，通过向量相似度（通常是 Cosine 相似度）来衡量相关性。

### 工作原理
1. 离线：用 Embedding 模型将所有文档转换为向量，存入向量数据库
2. 在线：将查询转换为向量，在向量空间中找最近邻

### 特点
- 优点：理解语义，能处理同义词、近义词，跨语言检索
- 缺点：对精确关键词匹配效果不如 BM25，计算开销大

### 常用 Embedding 模型
- **sentence-transformers**：开源，多语言支持
- **OpenAI text-embedding-3**：效果优秀，需要 API
- **BGE 系列**：中文效果好，智源开源

## 混合检索（Hybrid Search）

结合 BM25 和向量检索的优势，是当前 RAG 系统的最佳实践。

### RRF（Reciprocal Rank Fusion）
最常用的结果融合方法：

```
RRF_score(d) = Σ 1 / (k + rank_i(d))
```

其中 k=60（经验值），rank_i(d) 是文档 d 在第 i 个检索结果列表中的排名。

### 加权融合
```
final_score = α × bm25_score + (1-α) × vector_score
```

α 通常取 0.5，可以根据任务调整。

### 为什么混合检索更好
- BM25 擅长：精确关键词、专有名词、代码、数字
- 向量检索擅长：语义理解、同义词、模糊查询
- 混合后两者互补，召回率和精度都更高

## 重排序（Reranking）

召回阶段拿到候选集后，用更精确（但更慢）的模型重新排序。

### Cross-Encoder
- 将 (query, document) 拼接后输入模型，直接输出相关性分数
- 比 Bi-Encoder（向量检索）精度高，但不能预计算，速度慢
- 适合对 top-100 候选集重排

### 常用重排序模型
- `cross-encoder/ms-marco-MiniLM-L-6-v2`（英文）
- `BAAI/bge-reranker-large`（中英文）

### 两阶段检索流程
```
向量检索 → top-100 候选 → Cross-Encoder 重排 → top-5 最终结果
```

## 学习排序（Learning to Rank）

用机器学习模型来学习排序函数，常用于传统搜索引擎。

### 三种范式
- **Pointwise**：对每个文档独立打分，转化为回归/分类问题
- **Pairwise**：学习两个文档的相对顺序（RankNet、LambdaMART）
- **Listwise**：直接优化整个排序列表（ListNet）

### 特征工程
传统 LTR 需要手工设计特征：
- 查询-文档特征：BM25 分数、TF-IDF、词频匹配数
- 文档特征：PageRank、文档长度、点击率
- 查询特征：查询长度、查询类型

## 现代搜索系统架构

```
用户查询
    ↓
查询理解（意图识别、实体抽取、查询改写）
    ↓
多路召回（BM25 + 向量检索 + 协同过滤 + ...）
    ↓
粗排（轻量模型，毫秒级）
    ↓
精排（复杂模型，Cross-Encoder 或大模型）
    ↓
业务规则（过滤、去重、多样性）
    ↓
最终结果
```

## 评估指标

| 指标 | 说明 |
|------|------|
| Precision@K | 前 K 个结果中相关文档的比例 |
| Recall@K | 前 K 个结果召回了多少相关文档 |
| MRR（Mean Reciprocal Rank）| 第一个相关结果排名的倒数均值 |
| NDCG（Normalized DCG）| 考虑相关性程度和位置的综合指标，最常用 |
| MAP（Mean Average Precision）| 多个查询的平均精度均值 |

## RAG 中的检索优化

在 RAG 场景下，检索质量直接决定生成质量：

1. **查询改写**：用 LLM 将口语化问题改写为检索友好的形式
2. **HyDE**：生成假设答案后用答案做检索，效果往往优于原始查询
3. **多路召回**：同时用多个查询或多种检索方式，取并集
4. **上下文压缩**：只返回检索文档中最相关的句子，减少噪声

# 协同过滤推荐算法

## 核心定义
基于用户历史行为数据，利用群体协作信息预测用户兴趣的推荐技术。无需分析物品内容特征，仅依赖“人”或“物”的关联关系进行推理。

## 主要类型
- **基于用户 (UserCF)**：发现兴趣相似的用户，推荐其喜欢的物品。适合社交网络及兴趣导向场景。
- **基于物品 (ItemCF)**：发现与用户已选物品相似的其他物品。稳定性更高，广泛应用于电商。

## 实现原理
1. **数据构建**：建立用户 - 物品评分矩阵。
2. **相似度计算**：使用余弦相似度或皮尔逊系数衡量用户/物品间距离。
3. **预测与排序**：加权计算预测评分，输出 Top-N 推荐列表。

## 关键挑战
- **冷启动**：新用户或新物品缺乏数据，无法计算相似度。
- **数据稀疏**：评分矩阵稀疏导致推荐精度下降。
- **扩展性**：大规模数据下计算开销大，需增量更新机制。

# AI 搜索融合算法技术原理

## 1. 核心定义
AI 搜索融合旨在整合多路检索策略（如关键词、向量）与模型优化技术，以平衡召回率与准确率，解决单一搜索模式的局限性。

## 2. 关键技术原理
- **混合搜索评分 (RRF)**
  - **原理**：Reciprocal Rank Fusion。将不同排序列表映射为倒数秩并加权聚合，消除量纲差异。
  - **价值**：融合关键词（BM25）与语义向量检索，弥补单一模态盲区。
- **算子融合**
  - **原理**：将计算图中的多个底层算子合并为单一内核执行。
  - **价值**：减少显存读写开销，降低推理延迟，提升 GPU 吞吐量。
- **GEO 优化**
  - **原理**：面向生成式引擎的内容结构化适配。
  - **价值**：提高内容被大模型理解、解析与引用的概率。

## 3. 技术对比
| 维度 | 传统搜索 | AI 融合搜索 |
| :--- | :--- | :--- |
| 匹配方式 | 关键词精确匹配 | 语义 + 关键词混合 |
| 性能瓶颈 | 索引扩展性 | 推理算力与融合延迟 |

# BM25 算法原理

## 核心定义
BM25（Best Matching 25）是信息检索中的概率排序模型，用于计算查询与文档的相关性得分。它是 TF-IDF 算法的改进版本。

## 工作原理
- **词频饱和**：引入参数 $k_1$，使词频增加带来的收益呈饱和曲线，避免重复词过度主导评分。
- **长度归一化**：引入参数 $b$，根据文档长度调整得分，防止长文档因包含更多词而获得虚高分数。
- **IDF 加权**：保留逆文档频率机制，降低常见词权重，提升稀有词重要性。

## 与 TF-IDF 对比
| 特性 | TF-IDF | BM25 |
| :--- | :--- | :--- |
| 词频处理 | 线性增长 | 饱和曲线 |
| 长度偏差 | 无修正 | 有归一化 |
| 适用性 | 基础检索 | 复杂检索/RAG |

## 应用场景
常用于搜索引擎、RAG（检索增强生成）系统中的混合检索，配合向量检索提升召回精度。

# BM25 算法与分词技术解析

### BM25 核心原理
*   **定义**：BM25 是一种基于概率检索模型的排序函数，主要用于评估文档与查询词的相关性。
*   **改进机制**：相比传统 TF-IDF，BM25 引入了**词频饱和度**和**文档长度归一化**。
    *   **词频饱和**：通过参数 $k_1$ 限制单个词项权重随词频无限增长，呈现边际效应递减。
    *   **长度归一**：通过参数 $b$ 调节文档长度对权重的影响，避免长文档因词频高而获得不合理的高分。

### 中英文分词方式对比
*   **英文分词**：
    *   **方式**：基于空格和标点符号进行切分，实现相对简单。
    *   **处理**：通常需配合 **Stemming**（词干提取）和 **Stopwords Removal**（停用词去除）来优化索引质量。
*   **中文分词**：
    *   **方式**：中文无天然空格分隔，需依赖特定分词器（如 Jieba, IK, HanLP）。
    *   **粒度挑战**：面临“词粒度”与“字粒度”的选择。词粒度语义更准确但易受未登录词（OOV）影响；字粒度召回率高但索引体积大，语义相对松散。
    *   **歧义问题**：需处理交叉歧义和组合歧义，分词质量直接影响 BM25 的检索效果。

### 优缺点分析
*   **优点**：
    *   算法成熟，计算效率高，适合大规模稀疏检索。
    *   对精确匹配和长尾词查询效果优异，可解释性强。
*   **缺点**：
    *   **语义缺失**：基于词汇重合度，无法理解同义词或上下文语义（即“词汇鸿沟”问题）。
    *   **分词依赖**：中文检索效果高度依赖分词器的准确性，分词错误会导致检索失效。

# 分词技术原理与 BM25 应用对比

### 核心概念
**Tokenization（分词）** 是将连续文本序列切分为离散语义单元的过程。它是 BM25 检索算法的前置步骤，分词粒度直接决定倒排索引的结构与检索效果。

### 中英文分词差异
*   **英文：** 天然具备空格分隔，主要难点在于词干提取与形态还原。
*   **中文：** 无天然分隔符，需解决**歧义切分**（如“南京市/长江大桥” vs “南京市长/江大桥”）与**未登录词**（OOV）识别问题。

### 主流分词方法优缺点对比

| 方法 | 原理 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| **Whitespace** | 按空格切分 | 速度快，实现简单 | 无法处理中文；无法解决英文形态变化 |
| **Jieba** | 词典匹配 + HMM 隐马尔可夫模型 | 中文语义完整，适合通用场景；支持自定义词典 | 依赖词典，对新词识别差；切分粒度不可逆 |
| **N-gram** | 滑动窗口切分字符片段 | 彻底解决 OOV 问题，模糊匹配能力强 | 语义理解弱；索引体积大，计算成本高 |

### BM25 检索影响
BM25 算法依赖词项频率（TF）和逆文档频率（IDF）。
*   **粒度过粗**（如整句索引）：导致召回率下降，无法匹配局部关键词。
*   **粒度过细**（如字粒度）：导致精确率下降，干扰词增多。
*   **最佳实践**：中文检索通常采用 Jieba 精确模式配合自定义停用词表，必要时结合 N-gram 提升容错性。

### 英文分词：Whitespace Tokenizer

*   **原理**：按空格字符直接切分文本。
*   **优点**：实现简单，计算开销极低，适用于大多数英文场景。
*   **缺点**：无法处理标点符号粘连（如 "word,"），无法识别复合词语义，易导致词汇表膨胀。

### 中文分词工具对比

**1. Jieba (结巴分词)**
*   **原理**：基于前缀词典构建有向无环图（DAG）计算最大概率路径，结合 HMM 模型识别未登录词。
*   **优点**：支持多种分词模式（精确、全模式、搜索引擎模式），社区活跃，易用性高。
*   **缺点**：未登录词识别准确率一般，词典需手动维护，长文本处理速度较慢。

**2. THULAC (THU Lexical Analyzer)**
*   **原理**：基于结构化感知器与在线训练算法。
*   **优点**：分词准确率高，处理速度快（优于 Jieba），集成词性标注功能。
*   **缺点**：模型文件较大，自定义词典灵活性不如 Jieba。

### BM25 与分词的关系

*   **核心影响**：BM25 算法依赖 Term 频率（TF）和逆文档频率（IDF），分词粒度直接决定搜索相关性。
*   **粒度权衡**：
    *   **粗粒度**：召回率降低，但精确度高。
    *   **细粒度**：召回率提高，但可能引入噪声。
*   **优化建议**：中文搜索通常需配合停用词过滤，并根据业务场景（如电商、新闻）选择合适的分词器与自定义词典。

# BM25 分词机制与 Tokenizer 对比

### 1. BM25 核心原理
BM25 是一种基于概率检索模型的排序算法，改进自 TF-IDF。其核心通过引入**文档长度归一化因子**，解决了长文档因词频高而排名虚高的问题，使短文档中的关键词权重更合理。

### 2. 中英文分词差异
*   **英文分词**：基于空格和标点符号进行切分，处理相对简单。重点在于**词干提取**（Stemming，如将 running 归约为 run）和停用词过滤，以减少词形变化对匹配的影响。
*   **中文分词**：由于中文无天然空格分隔，需依赖分词器（如 Jieba、IK Analyzer）。核心难点在于**歧义消除**和**未登录词**（OOV）处理。常用策略包括最大匹配法和统计模型（CRF/HMM）。

### 3. Tokenizer 类型与对比
*   **传统搜索分词**：基于词典匹配（如 Lucene StandardTokenizer）。优点是精确匹配度高，适合关键词检索；缺点是依赖词典更新，难以处理新词。
*   **LLM Tokenizer**：基于 Subword 算法（如 BPE、WordPiece）。优点是词表固定、无 OOV 问题。但在 RAG 系统中，若检索端（BM25）与生成端分词标准不一致，会导致**词元不匹配**（Token Mismatch），降低召回率。

### 4. 优缺点总结
*   **优点**：计算高效、可解释性强、对精确关键词检索效果优异。
*   **缺点**：缺乏语义理解（无法处理同义词替换）、对分词质量高度敏感、长尾词召回困难。

### BM25 与分词器技术解析

**BM25 核心原理**
*   **定义**：BM25 是一种基于概率检索模型的排序函数，常用于搜索引擎相关性评分，是 TF-IDF 的改进版。
*   **机制**：通过引入参数 $k_1$ 控制词频饱和度（防止某个词出现次数过多主导得分），参数 $b$ 控制文档长度归一化（惩罚过长文档）。
*   **核心公式**：$Score(Q,D) = \sum_{i} IDF(q_i) \cdot \frac{f(q_i, D) \cdot (k_1 + 1)}{f(q_i, D) + k_1 \cdot (1 - b + b \cdot \frac{|D|}{avgdl})}$。

**中英文分词方式对比**
*   **英文分词**：
    *   **特点**：以空格为天然分隔符，分词相对简单。
    *   **处理**：需进行词干提取和词形还原，常用 Standard Tokenizer。
*   **中文分词**：
    *   **特点**：中文无天然空格，分词是核心难点。
    *   **挑战**：需解决歧义切分（如“南京市长江大桥”）和未登录词（OOV）识别问题。
    *   **常用工具**：Jieba（结巴分词）、IK Analyzer、HanLP。

**分词器优缺点对比**
*   **词典分词**：
    *   *优点*：速度快，可控性强，适合特定领域。
    *   *缺点*：无法识别新词，维护成本高。
*   **统计/机器学习分词**：
    *   *优点*：能识别新词，泛化能力强。
    *   *缺点*：计算开销大，需大量标注数据。
*   **对 BM25 的影响**：分词粒度直接影响倒排索引结构。粒度过粗会降低召回率，过细则降低准确率。在 RAG（检索增强生成）系统中，分词策略需与 Query 处理保持一致。

### 英文分词技术对比：Whitespace、NGram 与 Stemmer

#### 1. Whitespace Tokenizer（空白符分词器）
*   **定义**：最基础的分词方式，仅根据空白字符（空格、制表符、换行符）切分文本。
*   **优点**：
    *   **简单高效**：算法复杂度低，处理速度快。
    *   **保留原貌**：不破坏单词结构，适合精确匹配场景。
*   **缺点**：
    *   **标点干扰**：无法分离附着在单词上的标点符号（如 "word," 会被视为一个整体）。
    *   **语义缺失**：无法处理复数、时态等形态变化，召回率较低。

#### 2. NGram Tokenizer（N-gram 分词器）
*   **定义**：将文本切分为固定长度的字符序列。例如，"quick" 的 2-gram（bigram）切分为 "qu", "ui", "ic", "ck"。
*   **优点**：
    *   **模糊匹配**：对拼写错误、前缀搜索有极强的容错能力。
    *   **语言无关**：不依赖词典，适用于任何语言。
*   **缺点**：
    *   **索引膨胀**：生成的 Token 数量巨大，导致索引体积显著增加。
    *   **精度较低**：容易产生误匹配，缺乏语义理解。

#### 3. Stemmer（词干提取器）
*   **定义**：通常作为 Token Filter 使用，通过算法移除单词的后缀，将其还原为词干形式（如 "running" -> "run"）。
*   **优点**：
    *   **提升召回率**：能将同一语义的不同形态单词归一化。
    *   **缩减索引**：减少了存储的词汇量。
*   **缺点**：
    *   **过度提取**：可能将不同含义的词错误合并（如 "university" 和 "universe" 可能都被处理为 "univers"）。
    *   **结果非词**：提取出的词干可能不是合法的单词。
