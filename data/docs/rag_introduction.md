# RAG（检索增强生成）技术详解

## 什么是 RAG

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索与大语言模型生成能力结合的技术框架。它解决了大语言模型的核心局限：知识截止日期和幻觉问题。

RAG 的核心思路是：在生成回答之前，先从外部知识库中检索与问题相关的文档片段，然后将这些片段作为上下文（Context）提供给 LLM，引导模型基于真实信息生成回答。

## RAG 的工作流程

### 离线阶段（索引构建）
1. **文档加载**：从各种来源（PDF、网页、数据库）加载原始文档
2. **文本分块（Chunking）**：将长文档切割成固定大小的文本块
3. **向量化（Embedding）**：用 Embedding 模型将文本块转换为高维向量
4. **存储**：将向量和原始文本存入向量数据库

### 在线阶段（检索与生成）
1. **查询向量化**：将用户问题转换为向量
2. **相似度检索**：在向量数据库中找到最相似的 top-k 个文本块
3. **Prompt 构造**：将检索到的文本块拼接到 Prompt 中
4. **LLM 生成**：模型基于检索内容生成最终回答

## Chunking 策略

Chunking 是 RAG 中最关键的工程决策之一，直接影响检索质量。

### 固定大小分块
最简单的方式，按字符数或 token 数切割：
- **chunk_size**：每块大小，通常 256~1024 tokens
- **chunk_overlap**：相邻块的重叠部分，通常为 chunk_size 的 10%~20%

重叠的目的是保留跨块的上下文信息，避免关键信息被切断。

### 语义分块
基于句子边界、段落、标题等语义单位切割，比固定大小更智能，但实现复杂。

### 选择建议
- 问答类任务：chunk_size=512，overlap=64
- 长文档摘要：chunk_size=1024，overlap=128
- 代码文档：按函数/类边界切割

## Embedding 模型选择

| 模型 | 维度 | 特点 |
|------|------|------|
| all-MiniLM-L6-v2 | 384 | 轻量，速度快，适合英文 |
| text-embedding-3-small | 1536 | OpenAI，效果好，需要 API |
| bge-large-zh | 1024 | 中文效果优秀，开源 |
| e5-large-v2 | 1024 | 多语言，效果均衡 |

## RAG 的优化方向

### 查询优化
- **查询改写**：用 LLM 将用户问题改写为更适合检索的形式
- **HyDE（假设文档嵌入）**：先让 LLM 生成一个假设答案，用假设答案做检索
- **多查询**：生成多个不同角度的查询，合并检索结果

### 检索优化
- **混合检索**：结合向量检索（语义）和 BM25（关键词），互补不足
- **重排序（Reranking）**：用 Cross-Encoder 对初步检索结果重新排序
- **父文档检索**：检索小块，返回大块，兼顾精度和上下文

### 生成优化
- **引用标注**：在回答中标注信息来源，提高可信度
- **答案验证**：检查生成答案是否有检索内容支撑

## RAG vs 微调

| 维度 | RAG | 微调 |
|------|-----|------|
| 知识更新 | 实时，更新文档即可 | 需要重新训练 |
| 成本 | 低 | 高 |
| 可解释性 | 高（可追溯来源） | 低 |
| 适用场景 | 知识库问答、文档检索 | 特定风格、领域适配 |

## 常见问题

**Q: RAG 和传统搜索引擎有什么区别？**
传统搜索返回文档链接，RAG 直接生成自然语言答案。RAG 理解语义，传统搜索依赖关键词匹配。

**Q: 向量检索找不到相关内容怎么办？**
可以结合 BM25 关键词检索（混合检索），或优化 Chunking 策略，或对查询进行改写。

**Q: 如何评估 RAG 效果？**
常用指标：Faithfulness（忠实度）、Answer Relevancy（答案相关性）、Context Recall（上下文召回率）。工具：RAGAS 框架。
