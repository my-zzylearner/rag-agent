# RAG（检索增强生成）技术详解

## 什么是 RAG

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索与大语言模型生成能力结合的技术框架。它解决了大语言模型的核心局限：知识截止日期和幻觉问题。

RAG 的核心思路是：在生成回答之前，先从外部知识库中检索与问题相关的文档片段，然后将这些片段作为上下文（Context）提供给 LLM，引导模型基于真实信息生成回答。

## RAG 的工作流程

### 离线阶段（索引构建）
1. **文档加载**：从各种来源（PDF、网页、数据库）加载原始文档
2. **文本分块（Chunking）**：将长文档切割成固定大小的文本块
3. **向量化（Embedding）**：用 Embedding 模型将文本块转换为高维向量
4. **存储**：将向量和原始文本存入向量数据库

### 在线阶段（检索与生成）
1. **查询向量化**：将用户问题转换为向量
2. **相似度检索**：在向量数据库中找到最相似的 top-k 个文本块
3. **Prompt 构造**：将检索到的文本块拼接到 Prompt 中
4. **LLM 生成**：模型基于检索内容生成最终回答

## Chunking 策略

Chunking 是 RAG 中最关键的工程决策之一，直接影响检索质量。

### 固定大小分块
最简单的方式，按字符数或 token 数切割：
- **chunk_size**：每块大小，通常 256~1024 tokens
- **chunk_overlap**：相邻块的重叠部分，通常为 chunk_size 的 10%~20%

重叠的目的是保留跨块的上下文信息，避免关键信息被切断。

### 语义分块
基于句子边界、段落、标题等语义单位切割，比固定大小更智能，但实现复杂。

### 选择建议
- 问答类任务：chunk_size=512，overlap=64
- 长文档摘要：chunk_size=1024，overlap=128
- 代码文档：按函数/类边界切割

## Embedding 模型选择

| 模型 | 维度 | 特点 |
|------|------|------|
| all-MiniLM-L6-v2 | 384 | 轻量，速度快，适合英文 |
| text-embedding-3-small | 1536 | OpenAI，效果好，需要 API |
| bge-large-zh | 1024 | 中文效果优秀，开源 |
| e5-large-v2 | 1024 | 多语言，效果均衡 |

## RAG 的优化方向

### 查询优化
- **查询改写**：用 LLM 将用户问题改写为更适合检索的形式
- **HyDE（假设文档嵌入）**：先让 LLM 生成一个假设答案，用假设答案做检索
- **多查询**：生成多个不同角度的查询，合并检索结果

### 检索优化
- **混合检索**：结合向量检索（语义）和 BM25（关键词），互补不足
- **重排序（Reranking）**：用 Cross-Encoder 对初步检索结果重新排序
- **父文档检索**：检索小块，返回大块，兼顾精度和上下文

### 生成优化
- **引用标注**：在回答中标注信息来源，提高可信度
- **答案验证**：检查生成答案是否有检索内容支撑

## RAG vs 微调

| 维度 | RAG | 微调 |
|------|-----|------|
| 知识更新 | 实时，更新文档即可 | 需要重新训练 |
| 成本 | 低 | 高 |
| 可解释性 | 高（可追溯来源） | 低 |
| 适用场景 | 知识库问答、文档检索 | 特定风格、领域适配 |

## 常见问题

**Q: RAG 和传统搜索引擎有什么区别？**
传统搜索返回文档链接，RAG 直接生成自然语言答案。RAG 理解语义，传统搜索依赖关键词匹配。

**Q: 向量检索找不到相关内容怎么办？**
可以结合 BM25 关键词检索（混合检索），或优化 Chunking 策略，或对查询进行改写。

**Q: 如何评估 RAG 效果？**
常用指标：Faithfulness（忠实度）、Answer Relevancy（答案相关性）、Context Recall（上下文召回率）。工具：RAGAS 框架。

# RAG 文本分块（Chunking）策略与技术原理

## 核心概念
将长文本分割为适合向量数据库存储的片段，是 RAG 系统检索精度与生成质量的基础环节。

## 主流分块策略
- **固定长度**：按字符或 Token 数切分，实现简单但易割裂语义。
- **递归分块**：基于段落、标题等自然分隔符递归切割，保留文档结构。
- **语义分块**：利用 NLP 模型识别语义边界，确保上下文逻辑完整。
- **父子分块**：父块用于检索定位，子块提供详细内容，平衡效率与精度。

## 关键技术点
- **重叠（Overlap）**：相邻块保留部分重复内容，防止关键信息在边界丢失。
- **粒度平衡**：块过大引入噪声，过小丢失上下文，需依场景调整。
- **领域适配**：针对代码、法律、医疗等不同文本类型定制分块规则。

## 选型建议
无通用最优解，建议结合具体数据集进行多策略对比测试。

# RAG 文档分块（Chunking）策略详解

## 1. 核心分块算法
- **固定长度分块**：按字符或 Token 数均匀切割。优点是处理速度快，缺点是易切断语义完整性。
- **递归字符分块**：优先按段落、句子等自然分隔符切分，不足时再按固定长度补全。兼顾了文档结构与灵活性。
- **语义分块**：基于 Embedding 相似度计算，在语义边界处切分。能最大程度保留上下文关联，但计算成本较高。

## 2. 重叠策略（Overlap）
- **原理**：相邻分块间保留部分重复内容，形成滑动窗口。
- **作用**：防止关键信息因切分而丢失，确保检索时上下文连贯，提升召回率。
- **配置**：通常设置为分块大小的 10%-20%，需结合模型上下文窗口调整。

## 3. 策略对比与选型
- **精度 vs 速度**：固定长度最快，语义分块最准。
- **适用场景**：结构化数据推荐递归分块；长文本深度理解推荐语义分块。
- **原则**：在检索精度与处理延迟间取得平衡。

# RAGAS 评估机制

### 定义
RAGAS (Retrieval Augmented Generation Assessment) 是一个专注于 RAG（检索增强生成）系统的自动化评估框架。它利用大语言模型（LLM）作为评判者，旨在替代传统的人工评估，实现对 RAG 流程的无参考、快速迭代评估。

### 核心评估指标
RAGAS 通过以下四个核心维度量化系统性能：

*   **忠实度**：衡量生成的答案是否完全源于检索到的上下文，用于检测模型“幻觉”。若答案中的声明无法在上下文中找到依据，则忠实度低。
*   **答案相关性**：衡量答案是否准确回应了原始问题。评估方式通常让 LLM 根据答案反向生成问题，计算生成问题与原问题的相似度。
*   **上下文精确度**：衡量检索到的上下文中是否包含无关信息。重点评估检索系统区分相关文档与噪声的能力。
*   **上下文召回率**：衡量检索内容是否覆盖了回答问题所需的所有背景信息，评估检索系统的全面性。

### 工作原理
该框架采用 **"LLM-in-the-loop"** 机制。通过精心设计的提示词，让高性能 LLM（如 GPT-4）对检索内容和生成答案进行推理、验证和打分，最终计算出各项指标的数值，帮助开发者快速定位检索或生成模块的问题。
